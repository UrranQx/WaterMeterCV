# WaterMeterCV HTTP Service

Справка для интеграторов: контракт HTTP-API, коды ошибок, ограничения, рекомендации по деплою.

Базовый URL: `http://<host>:8000` (порт настраивается через `WATERMETERCV_PORT`).

---

## `POST /predict`

Принимает изображение счётчика, возвращает распознанную строку цифр и усреднённый confidence.

- **Content-Type:** `multipart/form-data`
- **Поле:** `image` — файл JPEG или PNG
- **Лимит размера:** 10 MB (413 при превышении)

### Пример запроса

```bash
curl -X POST -F "image=@meter.jpg" http://localhost:8000/predict
```

### Ответ 200 OK

```json
{
  "digits": "00123456",
  "confidence": 0.87
}
```

| Поле | Тип | Описание |
|---|---|---|
| `digits` | `string` | Строка из цифр `0-9`. Ведущие нули сохраняются. Пустая строка, если ROI не найдена или OCR не дал детекций. |
| `confidence` | `float` | Среднее `conf` по digit-боксам выбранной ориентации. `0.0`, если детекций нет. Округляется до 4 знаков. |

Distance/точка в числе в ответе **не** возвращаются — сервис отдаёт только целую часть показаний счётчика.

### Коды ошибок

| Код | Причина | Тело |
|---|---|---|
| `400` | Пустая загрузка либо не удалось декодировать изображение (битый файл, неподдерживаемый формат) | `{"detail": "Failed to decode image"}` / `{"detail": "Empty upload"}` |
| `413` | Файл больше 10 MB | `{"detail": "Image exceeds 10 MB limit"}` |
| `422` | Отсутствует поле `image` в multipart-форме | FastAPI validation-error payload |
| `500` | Внутренняя ошибка (сломанный weight-файл, cuda OOM, и т.п.). Трейс не возвращается; только тип исключения | `{"detail": "Internal error: <ExceptionType>"}` |
| `503` | Пайплайн ещё не инициализирован (startup не завершён) | `{"detail": "Pipeline not ready"}` |

## `GET /healthz`

Liveness/readiness-проба. Возвращает `200` как только FastAPI готов принимать запросы. В Docker-образах используется как `HEALTHCHECK`.

```json
{"status": "ok"}
```

> Примечание: `healthz` отвечает `200` сразу, даже если pipeline ещё грузит модели. Для жёсткой readiness-проверки используйте `/predict` на фикстурной картинке, либо опрашивайте `/info` — он проходит через тот же pipeline-lifecycle.

---

## `GET /info`

Метаданные текущей сборки — имена директорий весов и device.

```json
{
  "roi_model": "wm_yolo_roi_20260412_230832",
  "ocr_model": "yolo11m_20260414_194809",
  "device": "cuda:0"
}
```

Удобно для sanity-check после деплоя («точно ли это GPU-образ», «какие веса запечены»).

## Рекомендации по интеграции

- **Таймауты.** CPU-инференс одного снимка на slim-образе `python:3.13-slim` — порядка 0.5–1.5 s на типовом CPU. GPU — десятки ms. Клиентский таймаут ставьте с запасом: 5 s для CPU, 2 s для GPU.
- **Формат изображений.** Принимаются любые форматы, которые понимает `cv2.imdecode` — JPEG, PNG, BMP, WebP. Рекомендация: JPEG с исходным разрешением счётчика без лишнего downscale. Сервис сам обрежет ROI.
- **Размер файла.** Лимит 10 MB (`_MAX_IMAGE_BYTES` в `service/app.py`). Чаще всего это 3–5 MP JPEG. При необходимости изменить — поменяйте константу и пересоберите образ.
- **Параллелизм.** Uvicorn запускается в одной worker-процессе; для более высокой пропускной способности на GPU — запускайте несколько реплик за reverse-proxy (nginx / traefik). На CPU можно поднять `uvicorn --workers N`, но будьте аккуратны с памятью: каждая реплика держит свои копии моделей.
- **Холодный старт.** При первом запросе после старта идёт lazy-инициализация YOLO-сессий (уже в lifespan, но CUDA kernels прогреваются на первой картинке). Ожидайте 1–3 s задержки на первом `/predict`.
- **Пустой ответ.** Если ROI-детектор не нашёл область счётчика, сервис возвращает `200` с `{"digits": "", "confidence": 0.0}`. Используйте порог на стороне клиента (например, `len(digits) >= 4 and confidence > 0.3`).

## Вне scope

- **Батча нет.** В текущей версии один запрос = одно изображение. Batched-эндпойнт планируется позже.
- **Auth / rate limit / TLS** не реализованы — это задача обратного прокси.
- **Debug-артефактов нет.** Сервис не возвращает crop'ы, bbox'ы и прочие промежуточные данные.

## Лицензия

Сервис и запечённые в Docker-образы веса — **AGPL-3.0**. Дополнительно:
ROI-веса обучены на датасете под **CC BY-NC-ND 4.0**, поэтому **сервис
нельзя использовать в коммерческих контекстах** без замены ROI-весов или
получения отдельного гранта. AGPL §13 (network use) требует от оператора
сервиса предоставлять пользователям исходный код — поместите ссылку на
репозиторий в развёрнутую инсталляцию. Подробности — `NOTICE.md` в корне
репозитория.
