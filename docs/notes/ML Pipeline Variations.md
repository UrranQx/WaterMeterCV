**Мы решили использовать такой способ (Решили не использовать VLM, так как это очень долгий инференс)**
> Задача разбивается на несколько:
	1. Детекция окошка с цифрами на счетчике
	2. Предобработка полученного окна для следующего шага (выпрямить, повернуть, обрезать)
	3. Преобразовать картинку с цифрами в строку вида 1234567,890

1. Одностадийная детекция цифр
	1. Считываем каждую цифру и разделитель как объект. И правильно компануем их на выходе
	2. Простота архитектуры
	3. Предположительно YOLO
2. 2+ Стадии
	1. Локализация окошка ROI (Region of Interest)
	2. Реализация OCR по ROI 
___
## Локализация ROI
1. Faster R‑CNN (Detectron2) с FPN
2. YOLOv3/v5/v8 только для «reading window»
3. Сегментация (U‑Net / DeepLab / HRNet)
## Реализация OCR по ROI
1. CRNN + CTC (CNN → BiLSTM/GRU → CTC)
	1. Классика deep‑OCR для последовательностей: CNN вытаскивает фичи по ширине, RNN/GRU моделирует последовательность, CTC декодирует строку без явной сегментации.
2. Transformer‑OCR (TrOCR, PARSeq, ABINet и подобные)
	1. CNN/ViT encoder + transformer/attention decoder, сейчас такие модели демонстрируют лучшие результаты на scene text
	2. тяжелее для мобилки; логично скорее как серверная часть или как teacher‑модель для дистилляции.
3. Простой CNN + CTC без RNN
4. Per‑digit классификатор (fixed‑slots)
	1. Если формат всегда, например, 7 целых + запятая + 3 дробных, можно
		1. нарезать окошко на N вертикальных слотов (по шаблону) и
		2. обучать один CNN‑классификатор на 0–9/запятая/пусто
	2. мало гибкости — разные модели счётчиков, разная ширина/позиция цифр.
