## 2) Fraction-aware правила (реализовано)

Для `value_text` в WM:

1. 3+ знака после запятой: оставить как есть.
2. 2 знака после запятой: добавить скрытый третий `0`.
3. 1 знак после запятой, если он НЕ `0`: трактовать как сокращенную дробь и добавить `00`.
4. Для `.0`:
- если в правой дробной зоне ROI красного нет -> считаем счетчик без дробной части (дробь убираем);
- если красный есть -> считаем, что дробные барабаны есть, восстанавливаем как `000`;
- если определить нельзя -> сохраняем исходный вид `.0`.

Проверенные примеры после регенерации:
- `id_609_value_269_85` -> `269850`
- `id_58_value_336_0` -> `336`
- `id_90_value_729_0` -> `729000`
- `id_1161_value_71_024` -> `71024`

## 5) Практический вывод на текущий момент

На данном этапе имеет смысл фокусироваться на связке:
- crop path: `wm_bbox` + `wm_polygon` (как контроль)
- OCR: `ultralytics_yolo11m_baseline`

Почему:
1. Это единственный стабильный и уже отлаженный pipeline в текущем окружении.
2. По текущим метрикам он проходит Stage A и дает лучший полезный результат.
3. Дополнительные OCR-движки сейчас дают либо плохую практическую пользу, либо runtime/infra-ограничения в среде.

Рекомендация:
- Пока не расширять семейство OCR-моделей.
- Сначала использовать новые failure-dumps для точечной донастройки и улучшения качества на ошибках.
